# Trace 产品升级实施路线图

日期：2026-07-13

依据：`docs/superpowers/specs/2026-07-13-trace-top-level-product-design.md`

## 1. 目的

本路线图把已经确认的 Trace 顶层设计拆成可以独立实施、独立测试和独立回退的阶段。它负责说明阶段边界、依赖关系和交付顺序；每一阶段开始前仍需编写单独的详细实施计划。

现有系统已经具备日报、工作台账、周报、月报、年报、配置、里程碑、知识资产和多格式导出基础。升级应尽量复用现有页面和 API，持续保持系统可用，不进行一次性整体重写。

## 2. 总体策略

- 先稳定数据，再扩展业务对象，最后重构报告与导出。
- 每个阶段都必须兼容现有 SQLite 数据。
- 数据库结构变化全部通过可重复执行的迁移完成。
- 新模型启用前先保留旧字段作为兼容快照，确认迁移正确后再决定是否移除。
- 每个阶段均以完整测试和生产构建通过为交付前提。
- 用户现有数据在迁移前必须可备份，迁移失败不得留下半完成状态。
- 不在同一阶段同时重构数据库、全部页面和全部报告口径。

## 3. 阶段划分

### 阶段 0：基线与数据保护

目标：建立升级前的可信基线。

交付：

- 确认现有完整测试、类型检查和生产构建通过。
- 记录当前 SQLite 表结构和数据数量。
- 验证 JSON 备份可以覆盖现有核心数据。
- 为数据库迁移建立临时数据库测试方式。
- 明确每次迁移前的备份和失败恢复步骤。

完成标准：

- 可以在临时数据库上重复运行结构初始化。
- 升级前后可以比较记录、配置、标准、里程碑和知识资产数量。
- 不修改任何现有业务口径。

### 阶段 1A：迁移框架、标准版本与系数快照

目标：先解决历史工作当量可能随标准变化而失真的问题。

交付：

- 数据库迁移版本表和事务化迁移执行器。
- 工作当量标准版本对象。
- 现有标准迁入“历史标准”版本。
- 标准按版本维护和匹配。
- 工作记录保存计量单位、系数来源、标准 ID、标准版本 ID 和公式版本。
- 旧记录标记为历史来源，不推测当时的具体标准。
- 日报展示自动精确匹配、通用规则、手动填写和历史数据等来源。
- 配置页可以查看、创建和启用标准版本。

详细计划：

`docs/superpowers/plans/2026-07-13-workload-provenance-foundation-implementation-plan.md`

### 阶段 1B：项目实体化

目标：把工作记录中的项目名称文本升级为可持续追踪的项目对象。

交付：

- 项目表、项目别名和项目状态。
- 从现有非空 `projectName` 去重生成项目。
- 工作记录增加 `projectId`，同时保留 `projectName` 兼容快照。
- 项目增改查、归档、重新启用和重复项目合并。
- 日报搜索、选择和快速创建项目。
- 项目详情展示投入、时间线和当前重点。
- 非项目事项有明确标记，不再依赖空字符串猜测。

完成标准：

- 所有现有项目名称均能映射或明确保留为未归并项。
- 项目改名不会改写历史记录标题或破坏报告。
- 旧报告在迁移后保持原有统计结果。

### 阶段 2：成果管理与知识资产迁移

状态：已完成（2026-07-14）。

目标：建立正式交付、问题解决、阶段进展和可复用资产的统一成果模型。

交付：

- 成果、成果状态历史、成果与记录关系、成果与能力关系。
- 从现有知识资产迁移为“可复用资产”成果。
- 迁移时忽略旧附件和外部链接字段，不继续提供新增入口。
- 支持从日报、台账和项目创建成果。
- 项目聚合成果，成果汇总关联工时和原始工作当量。
- 多成果共享同一记录时，跨成果总计按记录去重。

完成标准：

- 原有知识资产标题、摘要、来源记录和标签完整保留。
- 每项成果可以追溯到来源记录和项目。
- 成果状态符合“计划中、推进中、阶段成果、已完成”。

实际交付还包括成果按年份、项目、类型、状态、能力和关键词筛选，日报、台账与项目的预填入口，以及周报、月报、年报和 Office 导出的成果接入。

### 阶段 3：工作台账与数据质量

状态：已完成（2026-07-14）。

目标：把全部记录页升级为长期可维护的工作事实档案。

交付：

- 周、月、年和自定义周期筛选。
- 项目、业务、工作类型、产品、子任务、能力、系数来源和成果状态组合筛选。
- 紧凑列表、展开详情和筛选范围统计。
- 缺少项目、能力、工时、系数、内容等质量提醒。
- 重复分类、疑似重复项目和手动系数占比提醒。
- 从筛选结果创建成果或交给报告模块。
- 重要删除操作显示关联影响。

完成标准：

- 多年数据仍能快速定位。
- 所有质量提醒可以跳转到来源记录。
- 提醒不自动修改用户数据。

实际交付还包括筛选范围内的项目与成果去重统计、紧凑可展开记录行、筛选后选择一致性、多记录批量提炼成果、删除影响预览，以及桌面和移动端响应式台账布局。

### 阶段 4：成长目标、能力分配与里程碑

目标：把能力成长和职业目标建立在真实工作证据上。

交付：

- 结构化能力分配表。
- 旧能力字符串迁移为平均分配。
- 日报支持多能力选择和可选手动比例。
- 职业目标、公司培养目标和年度目标。
- 数量型、投入型、阶段型和持续型里程碑。
- 自动进度、人工修正原因和证据追溯。
- 能力停滞、目标偏离、临期逾期和高投入低产出提醒。
- 不生成个人成长总分。

完成标准：

- 一条记录的原始工时和工作当量始终只计算一次。
- 能力分配比例合计为 100%。
- 目标进度可以解释计算来源。

### 阶段 5：复盘、报告与行动型展板

目标：形成日事实、周执行、月复盘、年总结的完整层级。

交付：

- 周报手工复盘字段和草稿。
- 月报成绩、不足、原因、改进、成长与下月规划。
- 年报项目贡献、成果、能力、目标和下一年度规划。
- 数据视图与汇报视图。
- 周期对比、投入集中度、投入产出和下一步提醒。
- 所有统计可以下钻到来源记录。
- 年度汇报折算系数默认 100%，仅影响本次预览和导出。
- 原始明细保持完整，报告详情默认紧凑。

完成标准：

- 自动统计与手工总结明确区分。
- 传统业务不自动打折。
- 报告修正通过来源记录完成，不直接覆盖统计结果。

### 阶段 6：Excel 标准导入、年度导出与完整备份

目标：完成与公司工作当量表和长期数据安全相关的能力。

交付：

- 读取“技术部工作当量统计表-杨川.xlsx”的“工作当量标准”工作表。
- 导入预览、新增、重复、冲突和无效项识别。
- 用户确认后原子写入新标准版本。
- 年度 Excel 输出工作明细、项目、成果、能力和工作当量。
- 支持公司年度工作当量表结构。
- 完整备份包和恢复前预检。
- 按年度归档但不删除数据。

完成标准：

- Excel 导入失败不产生半完成版本。
- 恢复前能显示影响范围。
- 年度折算值与原始值同时可辨认，折算不写回数据库。

### 阶段 7：导航整合与年度验收

目标：把分阶段交付整合为一致、稳定的七板块产品。

交付：

- 导航统一为今日工作台、工作台账、项目管理、成果管理、成长与目标、复盘与汇报、配置与数据。
- 清理已经被新对象替代的旧入口和兼容界面。
- 更新 README、架构文档和需求边界。
- 对日常录入、项目复盘、成果沉淀、月报和年报执行完整流程验收。
- 使用真实年度数据副本验证统计与导出。

完成标准：

- 七个板块职责清晰且无重复入口。
- 关键流程在桌面和移动视口均可使用。
- 全量测试、类型检查、生产构建和导出验收全部通过。

## 4. 依赖关系

```text
阶段 0
  ↓
阶段 1A 标准与历史口径
  ↓
阶段 1B 项目实体
  ↓
阶段 2 成果管理
  ↓
阶段 3 工作台账
  ↓
阶段 4 成长与目标
  ↓
阶段 5 复盘与汇报
  ↓
阶段 6 导入、导出与备份
  ↓
阶段 7 整体验收
```

阶段 3 和阶段 4 可以在阶段 2 完成后并行设计，但实施时仍应分别验收。阶段 6 的 Excel 标准导入依赖阶段 1A 的标准版本模型，年度导出依赖阶段 2、4、5 的数据结构。

## 5. 每阶段固定验收

每个阶段完成前必须执行：

```powershell
pnpm run test
pnpm run typecheck
pnpm run build
```

涉及前端页面时，还需：

- 启动本地开发服务。
- 验证桌面和移动视口。
- 检查空状态、加载状态、错误状态和长文本。
- 验证新增、编辑、筛选、删除或归档的完整流程。

涉及数据库迁移时，还需：

- 在全新数据库上运行。
- 在旧结构临时数据库上运行。
- 重复运行迁移并确认幂等。
- 人为制造失败并确认事务回滚。
- 比较迁移前后的数据数量和关键合计。

## 6. 提交策略

- 每个任务形成一个可测试提交。
- 数据迁移、后端接口、前端接入和界面调整分别提交。
- 不把用户已有的无关工作区修改带入提交。
- 每阶段完成并通过验收后再推送共享。
- 不在未验证迁移和数据备份的情况下进入下一阶段。
