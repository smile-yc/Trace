# Trace 阶段 1B 项目实体化设计

日期：2026-07-14

依据：

- `docs/superpowers/specs/2026-07-13-trace-top-level-product-design.md`
- `docs/superpowers/plans/2026-07-13-trace-product-roadmap.md`

## 1. 目标

把工作记录中的项目名称文本升级为可持续追踪的项目对象，使项目成为工作记录、后续成果、成长证据和周期报告之间的稳定关联主线。

本阶段交付完整的个人项目闭环：历史迁移、项目维护、别名搜索、状态管理、归档、重新启用、重复项目合并、日报关联和项目投入复盘。项目管理仍然服务于个人工作量化，不扩展为团队任务分派、审批或协同系统。

## 2. 范围与边界

### 2.1 本阶段包含

- 项目对象、项目别名和记录项目关联。
- 历史 `projectName` 数据迁移。
- 项目创建、编辑、状态变更、归档、重新启用和合并。
- 日报中的项目搜索、选择、快速创建和明确非项目模式。
- 项目列表与详情中的投入、轨迹和当前重点。
- 现有周报、月报、年报和导出的兼容。

### 2.2 本阶段不包含

- 团队成员、任务分派、审批、风险和项目进度百分比。
- 附件和外部链接。
- 正式成果模型；项目详情只预留成果空状态，成果关系在阶段 2 实现。
- 项目投入价值评分或传统业务自动折扣。
- 项目永久删除。

## 3. 数据模型

### 3.1 项目

新增 `projects` 表，字段为：

- `id`
- `name`
- `normalizedName`
- `shortName`
- `status`：`planned`、`active`、`paused`、`completed`、`archived`
- `startDate`
- `endDate`
- `personalRole`
- `goal`
- `description`
- `completionSummary`
- `mergedIntoProjectId`
- `archiveTime`
- `createTime`
- `updateTime`

`mergedIntoProjectId` 只用于保留合并来源。被合并项目保留原始记录，状态改为 `archived`，不直接删除。

### 3.2 项目别名

新增 `project_aliases` 表，字段为：

- `id`
- `projectId`
- `alias`
- `normalizedAlias`
- `createTime`

项目简称、改名前名称和被合并项目名称均可成为别名。项目名称、简称和别名不得跨有效项目产生无法区分的冲突。

### 3.3 工作记录关联

`records` 新增：

- `projectId`：可空项目外键。
- `projectRelation`：`project`、`non_project`、`unassigned`。

三种状态含义：

- `project`：必须存在有效 `projectId`。
- `non_project`：必须满足 `projectId = null`，表示用户明确确认是非项目事项。
- `unassigned`：只用于迁移后的历史空项目记录；新记录不得以该状态保存。

原 `projectName` 字段继续保留为保存时的项目名称快照。项目改名和项目合并都不改写该字段。

## 4. 历史数据迁移

迁移通过现有版本化、事务化数据库迁移框架执行。

1. 读取所有非空 `records.projectName`，去除首尾空格。
2. 仅对处理后完全相同的名称自动去重并创建一个项目。
3. 自动创建的历史项目默认状态为 `active`。
4. 对应记录写入项目 `projectId` 和 `projectRelation = project`。
5. 空 `projectName` 记录写入 `projectId = null` 和 `projectRelation = unassigned`。
6. 不根据简称、大小写、错别字或相似文本自动合并项目。
7. 疑似重复名称在项目管理中提示，由用户手动合并。

迁移不得改变记录数、工时、原始工作当量、系数快照、标准、里程碑和知识资产数量。重复执行初始化不得创建重复项目或重复别名。

## 5. 项目操作规则

### 5.1 创建与编辑

- 项目名称必填，简称可选。
- 状态默认 `active`；允许创建为 `planned`。
- 结束日期不得早于开始日期。
- 改名时，旧名称自动写入别名。
- 业务方向和涉及产品不重复手工维护，从关联记录实时汇总。

### 5.2 状态、归档与重新启用

- 状态支持计划中、进行中、暂停、已完成和已归档。
- 已完成和已归档项目不进入日报默认候选，但可以搜索和查看。
- 归档不解除历史记录关系，也不改变报告统计。
- 已归档项目可以重新启用为进行中。
- 已经合并到其他项目的来源项目不能单独重新启用。

### 5.3 合并

项目合并分为影响预览和确认执行：

- 预览显示来源项目、目标项目、受影响记录数、工时和原始工作当量。
- 来源和目标不能相同。
- 目标不能是已经合并到其他项目的项目。
- 确认后在一个数据库事务中修改所有关联记录的 `projectId`。
- 记录原 `projectName` 快照保持不变。
- 来源名称、简称和别名迁移为目标项目别名。
- 来源项目改为归档并记录 `mergedIntoProjectId`。
- 任何别名冲突或数据写入失败都整体回滚。

## 6. 页面与交互

### 6.1 项目管理列表

启用现有“项目管理”导航入口。列表默认展示计划中、进行中和暂停项目，提供：

- 名称、简称和别名关键词搜索。
- 状态筛选和已归档项目查看。
- 状态、个人角色、累计工时、原始工作当量、最近活跃日期和当前重点。
- 创建、编辑、完成、归档、重新启用和合并入口。

页面采用紧凑工作台布局，不增加装饰性图表或嵌套卡片。

### 6.2 项目详情

项目详情展示：

- 项目目标、个人角色、起止日期、说明和结项总结。
- 记录数、活跃天数、累计工时和原始工作当量。
- 从记录聚合的业务方向、产品和能力投入。
- 根据近期记录归纳的当前工作重点。
- 按时间倒序排列的工作轨迹。
- 成果空状态，为阶段 2 的正式成果关系预留位置。

所有汇总均可追溯到来源记录。项目只显示原始工作当量，不自动应用业务折扣。

### 6.3 日报录入

项目字段改为“项目事项 / 非项目事项”模式切换：

- 项目事项使用支持名称、简称和别名搜索的项目选择器。
- 支持在选择器中快速创建项目。
- 已完成和已归档项目不在默认候选中，但搜索后可以选择。
- 新记录必须选择项目或明确选择非项目事项。
- 编辑历史 `unassigned` 记录时显示“历史未关联”，由用户补充项目或改为非项目事项。
- 草稿保存 `projectId`、项目关联模式和项目名称显示快照；恢复时重新验证项目状态。

## 7. 接口设计

后端提供以下能力：

- `GET /api/projects`：搜索、状态筛选和归档筛选。
- `GET /api/projects/:id`：项目基础信息。
- `GET /api/projects/:id/summary`：项目指标、分布和时间线。
- `POST /api/projects`：创建项目。
- `PATCH /api/projects/:id`：编辑项目及状态。
- `POST /api/projects/:id/archive`：归档项目。
- `POST /api/projects/:id/reactivate`：重新启用项目。
- `GET /api/projects/:id/merge-preview?targetId=...`：合并影响预览。
- `POST /api/projects/:id/merge`：事务化合并。

记录接口增加 `projectId` 和 `projectRelation`。当 `projectRelation = project` 时，后端根据 `projectId` 读取项目并写入当前名称快照，不信任客户端提交的 `projectName`。

## 8. 数据一致性与错误处理

后端负责校验：

- 项目名称、简称和别名冲突。
- 项目状态和日期范围。
- 项目关联状态与 `projectId` 的组合。
- 项目是否存在、是否已经合并、是否允许选择。
- 合并来源、目标和别名迁移是否合法。

主要错误使用稳定错误码并返回明确中文提示，包括：

- `PROJECT_NAME_CONFLICT`
- `PROJECT_ALIAS_CONFLICT`
- `PROJECT_NOT_FOUND`
- `PROJECT_INVALID_STATUS`
- `PROJECT_INVALID_DATE_RANGE`
- `PROJECT_RELATION_INVALID`
- `PROJECT_NOT_SELECTABLE`
- `PROJECT_MERGE_TARGET_INVALID`

数据质量检查可以提示历史未关联和疑似重复项目，但不得自动修改用户数据。

## 9. 测试与验收

### 9.1 自动测试

- 全新数据库迁移、现有数据库副本迁移、重复初始化和失败回滚。
- 完全同名迁移去重、空项目保持未关联、近似名称不自动合并。
- 项目 CRUD、状态变更、归档、重新启用、别名搜索和合并事务。
- 项目事项、非项目事项、历史未关联和项目名称快照。
- 项目汇总的记录数、活跃天数、工时和原始工作当量。
- 原周报、月报、年报和导出统计兼容。

### 9.2 浏览器验收

- 桌面和移动视口检查项目列表、详情、搜索和状态筛选。
- 完整验证快速创建、日报关联、编辑历史未关联、归档、重新启用和合并确认。
- 检查空状态、加载状态、错误状态和长项目名称。
- 确认项目管理入口可达，其他六个板块不发生导航回归。

### 9.3 完成门槛

```powershell
pnpm run test
pnpm run typecheck
pnpm run build
git diff --check
```

数据库迁移还必须在全新数据库和现有数据副本上分别运行两次，并比较迁移前后的核心数量与合计。

本阶段通过全部验收后形成经过验证的本地提交，不推送 GitHub。远程同步由用户在后续整体功能验证完成后统一确认。

## 10. 后续依赖

阶段 2 的成果对象通过 `projectId` 关联项目。阶段 3 的工作台账使用项目状态、项目关系和数据质量结果进行组合筛选。阶段 5 的月报和年报使用项目投入、结项总结和成果关系生成项目贡献材料。
