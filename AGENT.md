# Trace Agent 统一约束

本文件约束所有参与 Trace 开发、审查和集成的 agent。任何角色说明、任务提示或临时判断与本文件冲突时，先停止工作并向总协调者报告，不得自行放宽约束。

## 1. 项目使命

Trace 是面向个人职业发展的工作量化、成果管理与成长复盘系统。核心使命是将日常工作事实持续沉淀为可量化、有关系、可复盘、可汇报的长期职业证据。

权威文档按以下顺序解释：

1. `docs/superpowers/specs/2026-07-13-trace-top-level-product-design.md`
2. `docs/superpowers/specs/2026-07-13-trace-ui-design.md`
3. `docs/superpowers/plans/2026-07-13-trace-multi-agent-collaboration-plan.md`
4. 当前角色文件和任务简报

## 2. 产品边界

- 不增加团队项目管理、成员分工、任务派发或审批。
- 不增加考勤、绩效审批、个人排名或综合评分。
- 不保存附件和外部链接。
- 不扩展为通用笔记或知识库；原知识资产迁移为成果中的可复用资产。
- 不自动判断工作价值，不把高投入等同于高价值。
- 不以图表数量作为完成标准。
- 首轮不实现深色模式。

## 3. 数据不变量

- 一条工作记录只保存一次原始工时和原始工作当量。
- 原始工作当量等于数量乘以实际折算系数。
- 每条记录保存实际系数、系数来源和标准版本，标准变化不静默重算历史。
- 多能力默认平均分配，允许手动分配，比例合计必须为 `100%`。
- 能力分摊只服务成长分析，不重复增加公司工作量。
- 同一记录支撑多个成果时，各成果可分别引用，跨成果汇总必须按记录去重。
- 传统业务不自动打折。
- 年度汇报折算为本次报告的临时参数，默认 `100%`，不得写回原始数据。
- 汇总数字、图表和报告材料必须能够返回来源记录。

## 4. UI 不变量

- 视觉方向为“冷静专业工作台 + 汇报级阅读体验”。
- 品牌色为 `#176B68`，悬停色为 `#125A57`，选中背景为 `#E7F2F0`。
- 普通圆角为 `6px` 至 `8px`，少阴影，无渐变，不使用卡片嵌套。
- 桌面侧边栏宽度为 `216px`；移动端使用顶部栏和抽屉导航。
- 每个页面标题区最多一个主要按钮。
- 长选项必须支持搜索；工作类型、产品、子任务必须联动筛选。
- 日报和报告草稿手动保存，不采用自动保存。
- 项目、成果默认使用紧凑列表，不默认使用看板。
- 不使用雷达图、排行榜、综合绩效分、装饰性仪表盘或大面积饼图。
- 能力投入图使用时间型气泡图：横轴时间、纵轴能力、面积表示分摊工作当量、颜色深浅表示分摊工时、特殊描边表示成果或里程碑。
- 所有桌面和移动端界面不得出现文字、按钮、表格、图表或浮层相互遮挡。

## 5. 工程规则

- 实施功能或修复缺陷时严格使用测试驱动：先写失败测试，确认失败原因，再写最小实现。
- 遵循现有 React、TypeScript、Express 和 SQLite 技术栈，不未经批准引入新框架。
- 使用 Lucide React 图标，不手绘已有标准图标。
- 结构化数据使用明确类型和解析器，不依赖脆弱字符串拼接。
- 文件应按职责拆分，避免继续扩大 `App.tsx`、`styles.css`、`database.ts` 和 `index.ts` 的职责。
- 不执行无关重构，不改写未分配给自己的业务域。
- 不修改或提交根目录 `.gitignore`；该文件包含用户未提交改动。
- 不撤销、覆盖或格式化其他 agent 或用户的修改。
- 禁止使用 `git reset --hard`、`git checkout --` 或破坏性清理命令。
- agent 不自行合并、变基或推送分支；由总协调者和集成 agent 处理。

## 6. 文件所有权

- Agent 1：数据基础、工作记录契约、标准版本、工作当量、配置后端和导入接口。
- Agent 2：设计变量、基础组件、应用外壳、导航和响应式布局。
- Agent 3：今日工作台、工作台账、项目、成果和配置数据页面。
- Agent 4：成长、目标、周月年报、图表、打印和导出。
- Agent 5：共享入口接线、合并冲突、跨域检查和最终验收。

角色不得修改禁止目录。确需修改共享文件时，在交接报告中列出接线需求，由 Agent 5 处理。

## 7. 测试与验收

每个 agent 先运行自己的定向测试，再运行受影响包的类型检查和构建。最终集成必须运行：

```powershell
pnpm run test
pnpm run typecheck
pnpm run build
```

前端改动还必须在真实浏览器中检查桌面和移动端；报告改动必须检查 A4 打印；数据改动必须验证旧数据兼容和历史口径稳定。

禁止通过删除断言、跳过测试、降低类型约束或隐藏错误来通过门禁。

## 8. Agent 交接格式

每次交付必须包含：

1. 状态：`DONE`、`DONE_WITH_CONCERNS`、`NEEDS_CONTEXT` 或 `BLOCKED`。
2. 提交号和提交标题。
3. 实际修改文件。
4. 新增或变更的数据结构、接口和页面入口。
5. 测试驱动证据：失败测试、通过测试和最终命令结果。
6. 需要 Agent 5 处理的共享接线点。
7. 已知限制、风险和未完成内容。

仅回复“已完成”不构成交付。
